# 规则

1. **改口只留终版**：遇到"不对""我是说""啊不是""等等"或同一句话重复但措辞不同，只保留最后确认的表达。

2. **删噪保意**：删除填充词（嗯、呃、那个、就是、然后、对对对、OK OK、怎么说呢）和结巴重复。但当"其实"承载实际转折语义时保留。

3. **弱化词精简**：删除或改写口语弱化词——"一下"直接删除；"可能""好像""大概"在表推测时改为"可能"一词保留，纯口癖时删除；"我觉得""我感觉"删除，直接陈述观点；句末"嘛""吧""啊"删除。

4. **人称精简**：祈使/指令句删除"你"；观点句删除"我觉得""我认为"直接陈述；"我们"在泛指时删除，在区分团队归属时保留。省略后若导致无主句或指代不清，必须保留必要主语。

5. **同义去重**：同一含义出现多次（换词重述、举例重述、强调性重复），只保留信息量最大、表达最精准的一种说法。

6. **口语书面化（仅正式语境）**：在正式汇报/技术文档语境下：搞→处理、搞定→完成、不做→不实施、搞不定→无法完成、没法→无法、做一轮→进行一轮。日常对话中保留口语原词，不做替换。

7. **连接词分级**：并列项间的"然后"直接删；话题转换的"然后"替换为"另外"；因果/时序关系保留或替换为更精确的连接词。

8. **语序重组**：先结果后原因 → 调整为先因后果；先请求后理由 → 调整为先理由后请求；散乱补充 → 整合到逻辑位置。

9. **结构化判断**：3 项及以上并列 → 编号列表；仅 2 项 → 用"并""和""同时"等自然连接，不用列表。当原文含数量提示词（"几个""三点""以下""第一第二"）且有并列内容 → 强制编号列表。话题切换处分段。

10. **数字规范**：口语数字一律转阿拉伯数字+单位。"三五个"="3-5 个"（范围）；"三点五个"="3.5 个"（确切值）；"十来个"="约 10 个"。"两三天"="2-3 天"。

11. **同音术语还原**：杰森→JSON、派森→Python、瑞迪斯→Redis、艾皮爱→API、西加加→C++、吉特→Git、多克→Docker。中英文间加空格，专有名词正确大小写。

# 示例

输入：嗯那个我想一下，这个接口应该返回一个列表，不对，应该返回一个分页对象，里面包含列表和总数
输出：这个接口应该返回一个分页对象，里面包含列表和总数。

输入：可以直接用那个命令行工具就可以了，反正就是不需要装额外的东西就可以跑起来
输出：直接用命令行工具即可，不需要装额外的东西。

输入：然后你把这个文件发给我然后再帮我看一下那个部署脚本然后顺便把日志清理一下
输出：请帮我：1. 把文件发给我；2. 检查部署脚本；3. 清理日志。

输入：页面加载特别慢我好像看了一下大概要三五秒才能打开可能是接口太慢了吧
输出：页面加载需要 3-5 秒才能打开，可能是接口响应太慢。

输入：这个项目我觉得大概要投个三五个亿吧，周期的话可能两三年能搞完
输出：这个项目预计投入 3-5 亿，周期约 2-3 年。

输入：你先把那个配置文件改一下然后重启一下服务
输出：先修改配置文件，然后重启服务。

输入：这个方案好像有两个问题吧，一个是性能不够好，还有一个是扩展性比较差
输出：这个方案有两个问题：性能不够好，且扩展性较差。

输入：我觉得这个功能需要加一个缓存，就是加一个缓存层，用瑞迪斯那个，就是一个缓存中间件，放在数据库查询前面，减少数据库压力，就是这样可以提升性能
输出：这个功能需要在数据库查询前加一个 Redis 缓存中间件，减少数据库压力，提升性能。

输入：我们选了方案 B，因为方案 A 成本太高了而且交付周期太长了还有就是技术风险也比较大
输出：由于方案 A 成本过高、交付周期过长且技术风险较大，因此选择了方案 B。

输入：这个项目进度太慢了，就是推进得不行，整个节奏完全跟不上
输出：该项目推进节奏严重滞后。

输入：今天晚上等会儿去吃火锅吧感觉挺久没吃了
输出：今天晚上等会儿去吃火锅吧，感觉挺久没吃了。

输入：The uh the deployment failed again I think it's because of the the docker configuration issue
输出：The deployment failed again. I think it's because of the Docker configuration issue.

输入：好的收到
输出：好的，收到。

输入：我觉得还行吧
输出：我觉得还行吧。
